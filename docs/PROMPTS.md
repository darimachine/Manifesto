## Prompt 1: Анализ на scope и редукция (D-10)

### Контекст
Имах оригинален план за 16 таблици и 5 генератора, но реално оставаше 1 седмица до изпит. Трябваше agressive scope cut.

### Prompt
```
Анализирай docs/PRODUCT_REQUIREMENTS.md и docs/DATABASE_SCHEMA.md.
Имам 1 седмица до изпит, един човек съм. Кои entities и features са MUST за core
концепцията на проекта (декларирай инфраструктура → генерирай compose), и кои
могат да отпаднат без да нарани историята?
Дай ми scope-cut решение което мога да защитя пред преподавателя.
```

### Отговор (резюме)
- Cut: HW/Hypervisor/VM таблиците (метаданни, не са в Docker layer)
- Cut: InfraRole/InfraUser (orthogonal към основната хипотеза)
- Cut: Network/ServiceNetwork (compose използва default network)
- Cut: Dockerfile/vhost/README генератори (60% от работата за 5% от value)
- Cut: JSON import/export (parser работа, не е core)
- Keep: 9 таблици, docker-compose + .env + Emmet генератори, manual health check stretch

### Оценка
✅ **Точен и полезен.** Решението е консервативно — отрязва workload не visible value. Записах като D-10 в `docs/DECISIONS.md`. Защитима е история: „Фокусирах scope върху Docker layer; HW/VM нивата са оставени като future work."

## Prompt 2: Stateless YAML генератор без външни библиотеки

### Контекст
Курсът забранява vendor зависимости. Трябваше си напиша YAML emitter on hand.

### Prompt
```
Напиши Manifesto\Services\DockerComposeGenerator на vanilla PHP 8.
Stateless клас, метод generate(int $projectId): string.
Ползва ProjectRepository + ServiceChildrenRepository (вече съществуват).
Output: валиден Docker Compose v3.8 YAML, който трябва да минава
"docker compose -f x.yml config" без warnings.
Правила: пропускай празни ports/env/volumes секции;
double-quote стойности с : # ' или whitespace; null env value → KEY:.
Без yaml_emit() extension. Без външни libs.
```

### Отговор (резюме)
Класът използва array of lines + `implode("\n", $lines)`. Има отделен helper `yamlScalar()` който проверява дали стойността има спец символи и я обвива в `"..."` ако да, escape-ва внутрешните `\` и `"`. Service name се валидира regex `/^[a-zA-Z0-9_-]+$/`. Емитира `# Generated by Manifesto · {project} · {ISO date}` header.

### Оценка
✅ **Много добър.** Кодът е чист, добре документиран. Ще тествам с реален `docker compose config` команда — ако минава без warnings, идеален. Една грешка в първата итерация: пропусна да escape-не backslash, поправих в follow-up.

## Prompt 3: Sub-folder URL detection без конфигурация

### Контекст
Лекторът ще пусне проекта в `http://localhost/w26/<unique>/` — НЕ root-а на Apache. Всички URL-и трябва да включват префикса автоматично.

### Prompt
```
Имам vanilla PHP front controller (public/index.php) зад Apache mod_rewrite.
Проектът трябва да работи и от document root, и от поддиректория
като /w26/manifesto-fn1/ — БЕЗ конфигурация в .env. Как да детектирам
base path-а от $_SERVER за да генерирам правилни линкове?
Покажи Request клас с basePath() метод и url() helper.
```

### Отговор (резюме)
Парсва `$_SERVER['SCRIPT_NAME']` (винаги абсолютен спрямо DocumentRoot), взема dirname-а, ако завършва на `/public` — отрязва това. Резултатът е base prefix-а (`''` за root, `/w26/manifesto-fn1` за поддиректория). `url('/login')` връща `<base>/login`.

### Оценка
✅ **Точен.** Работи и с Docker (root) и с XAMPP в поддиректория без никаква конфигурация. Използвам го в production кода — `Request::basePath()` в `src/Core/Request.php`.

## Prompt 4: Dynamic form rows за parallel arrays в PHP

### Контекст
Service форми имат ports/envs/volumes секции с N редове. Трябва vanilla JS за +Add row.

### Prompt
```
PHP form с parallel array names: ports[host_port][], ports[container_port][], 
ports[protocol][]. Искам vanilla JS бутон "+ Add port" който добавя нов ред 
с празни input-и. И × бутон на всеки ред да го премахва.
Без jQuery, без framework. Добави и checkbox "is_secret" за env vars,
който trябва да work-ва в parallel array (gotcha: unchecked checkbox не 
изпраща нищо в POST).
```

### Отговор (резюме)
Документира трик-а с checkbox: hidden field с същото name преди checkbox-а, който държи `0`/`1` toggle-нат от `onchange` handler-а на чекбокса. Така parallel array-ите остават синхронизирани независимо дали checkbox-ът е чекнат.

```html
<input type="hidden" name="envs[is_secret][]" value="0" class="env-secret-hidden">
<input type="checkbox" onchange="this.previousElementSibling.value = this.checked ? '1' : '0'"> secret
```

### Оценка
✅ **Брилиантен трик.** Без него parallel arrays биха се desync-нали (checkbox row 2 е unchecked → POST пропуска индекса → row 3 става row 2 в backend-а → коруптни данни). Това е истинска WTF gotcha която bъдещи разработчици ще запомнят.

## Prompt 5: Многослоен паралелен агентен workflow

### Контекст
Имах 4 независими блока работа (generators / controller / views / docker setup) и 1 ден до края.

### Prompt
```
Имам PHP проект на финалния етап. Създай 4 паралелни Sonnet sub-агента,
всеки с конкретен scope:
1. Создай 3 stateless generators в src/Services/
2. Создай GenerationController + GeneratedFileRepository
3. Создай 4 view файла (services/create+edit, projects/files+emmet)
4. Създай Dockerfile + docker-compose.yml

За всеки агент дай: пълен контекст на existing code patterns, exact file paths,
namespace conventions, repository signatures (имена на методи, типове).
Те трябва да work-ват независимо без да изчакват един друг.
```

### Отговор (резюме)
Параллелизмът работи — 4 файла бяха създадени за ~3 минути. Ключът е че всеки агент има пълен self-contained brief с pattern references. Нямаше merge conflicts защото всеки агент пише различни файлове.

### Оценка
✅ **Огромна productivity boost.** Без паралелизация би отнело 4× повече време. Lesson learned: parallelize when files don't overlap, give each agent enough context to work without back-and-forth.

## Какво НЕ работи добре с AI

1. **Schema migrations** — AI често забравя ENUM constraints или забравя `ON DELETE CASCADE`. Винаги review-вам ръчно.
2. **Edge cases в YAML escaping** — backslash в стойност, null vs празен низ, multi-line strings. Първи опит обикновено пропуска нещо.
3. **Subtle SQL bugs** — мисли че SELECT ще върне обект, връща false. Винаги проверявам с `if ($row === false)` срещу `if (!$row)`.
4. **CSS responsive breakpoints** — AI пише първи опит десктоп-only. Изрично да поискам mobile поведение.
5. **Изпускане на CSRF tokens** — на всяка нова форма трябва да си напомням „add csrf_field()". Hooks-овете в settings.json-а помагат тук.

## Резюме

AI като pair programmer работи отлично за:
- Boilerplate (CRUD контролери, views, repository methods)
- Parallel независими блокове (4 агента работят 4× по-бързо)
- Конвенция-следване (само трябва да дам 1 reference файл и patterns се прихващат)
- Документация (README, USER_GUIDE)

Не работи добре за:
- Architectural decisions (тук трябва човешки контекст за business value vs effort)
- Subtle bugs в edge cases (винаги ревю)
- Креативен дизайн от нищо

**Финален извод:** AI = многоръковден pair programmer. Не code monkey, но и не senior arhitect. Знай какво искаш, проверявай резултата, използвай parallel когато може.
